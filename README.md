# pi-feature-flow

Pi package for turning a feature description into:
- a feature folder
- a master spec
- an execution plan
- dependency-aware tickets
- automatic next-ticket execution with registry tracking

## Commands

- `/plan-feature <feature>` — plan a feature from an existing spec document (creates execution plan + tickets)
- `/init-feature <feature>` — scaffold a feature folder with a stub spec file
- `/start-feature <feature>` — show status and start or resume the next ticket
- `/next-ticket <feature>` — pick and execute the next available ticket automatically
- `/ticket-done <feature>` — mark the current in-progress ticket as done
- `/ticket-blocked <feature>` — mark the current in-progress ticket as blocked
- `/ticket-needs-fix <feature>` — mark the current in-progress ticket as needs-fix and optionally retry
- `/ticket-status <feature>` — show current feature ticket progress
- `/ticket-validate <feature>` — validate spec files, dependencies, and ticket structure
- `/feature-cost <feature>` — show cost breakdown by ticket and phase

## Installation

```bash
pi install git:github.com/leandr0ck/pi-feature-flow
```

## Optional subagent integration

This package does not bundle or require `pi-subagents` in order to load.

If your Pi environment already provides the `subagent` tool, `pi-feature-flow` will prefer subagent delegation for planner/worker/reviewer flows.

If `subagent` is unavailable, the package still works and falls back to direct execution with standard Pi tools such as `read`, `write`, `edit`, and `bash`.

## What it creates

```text
./docs/technical-specs/
  <feature>/
    01-master-spec.md
    02-execution-plan.md      (generated by /plan-feature)
    03-ticket-registry.json
    04-feature-memory.md         (generated — shared memory using a fixed template)
    tickets/
      STK-001.md
      STK-002.md
      STK-001-tester-notes.md     (TDD only — written by tester phase)
      STK-001-reviewer-notes.md   (written by reviewer phase)
      STK-001-worker-context.md   (written by chief — used on retry / continuation)
      STK-001-handoff-log.md      (written incrementally by tester/worker/reviewer/chief)
      ...
```

Each ticket must use the required template shape and declare dependencies and editable files explicitly:

```md
# STK-002 — Add welcome email trigger

## Goal
Send the welcome email when onboarding is completed for the first time.

- Requires: STK-001
- Files: src/onboarding/complete.ts, tests/onboarding/complete.test.ts

## Implementation Notes
- Reuse the onboarding completion event.
- Guard against duplicate sends.

## Acceptance Criteria
- Completing onboarding sends one welcome email.
- Repeating completion does not send duplicates.
```

## Fixed output templates

The package now uses fixed markdown templates for all handoff artifacts so the model does not invent its own structure.

### Tester notes

```md
# Tester Notes — STK-001

## Tests written
- <file path>: <what it tests>

## Test guidelines followed
- <project-specific test conventions or patterns applied>

## Notes for worker
- <anything the worker must know: file paths, imports, test command>
```

### Reviewer notes

```md
# Reviewer Notes — STK-001

## Verdict
<APPROVED | NEEDS-FIX | BLOCKED>

## Findings
- <specific issue, or 'none'>

## Evidence
- <test output, files inspected, typecheck result>
```

### Worker context

```md
# Worker Context — STK-001

## Status
<APPROVED | NEEDS-FIX | BLOCKED>

## Files modified
- <path>: <what was done> [complete | partial | failed]

## Reviewer findings
- <issue or 'none'>

## Continuation notes
- <what the next attempt must know>
```

### Handoff log

```md
# Handoff Log — STK-001

## Tester
- Tests written: <files and scope>
- Test guidelines followed: <project conventions used>
- Risks / assumptions: <none | bullets>
- Notes for worker: <key context>

## Worker
- Files changed: <paths>
- Technical decisions: <none | bullets>
- Risks / tradeoffs: <none | bullets>
- Notes for reviewer/chief: <key context>

## Reviewer
- Verifications: <checks performed>
- Findings: <none | bullets>
- Residual risks: <none | bullets>
- Recommendation: <APPROVED | NEEDS-FIX | BLOCKED>

## Chief
- Promoted to feature memory: <none | bullets>
- Reusable patterns for future tickets: <none | bullets>
- Continuation advice: <none | bullets>
```

## How it works

Run:

```text
/plan-feature <feature>
```

The package will:
1. Read `01-master-spec.md` from the feature folder.
2. Generate the execution plan and dependency-aware ticket files.
3. Validate the resulting structure.
4. Start implementation automatically and advance ticket by ticket until it finishes or hits a blocked / needs-fix state.

## Configuration

Config file:
- `.pi/feature-flow.json`

### Minimal config

```json
{
  "specsRoot": "./docs/technical-specs"
}
```

### Recommended config

```json
{
  "specsRoot": "./docs/technical-specs",
  "tdd": false,
  "agents": {
    "planner": { "model": "anthropic/claude-sonnet-4" },
    "worker": { "model": "anthropic/claude-sonnet-4" },
    "reviewer": { "model": "openai/gpt-4.1" },
    "chief": { "model": "anthropic/claude-sonnet-4" }
  }
}
```

### Supported fields

| Field | Type | Default | Description |
|---|---|---|---|
| `specsRoot` | `string` | `"./docs/technical-specs"` | Root directory containing feature folders. |
| `tdd` | `boolean` | `false` | Adds TDD-oriented guidance to planning and execution prompts. |
| `agents.<role>.agent` | `string` | builtin | Subagent name to use for that role. |
| `agents.<role>.model` | `string` | unset | Preferred model for that role. |
| `agents.<role>.thinking` | `string` | unset | Preferred thinking level for that role. |
| `agents.<role>.skills` | `string[]` | unset | Skills to enable for that role. |

**Agent roles:** `planner` (generates execution plan + tickets), `tester` (reads the ticket and writes tests only, plus tester notes + handoff log), `worker` (runs tests, implements, and updates handoff log), `reviewer` (validates the implementation, and if needed may patch tests/implementation within ticket scope using TDD-first discipline, then writes reviewer notes + updates handoff log), `chief` (writes feature memory + worker context + final handoff log summary).

### Runtime model switching

Role-level config is not only prompt guidance anymore: the extension now applies `agents.<role>.model` and `agents.<role>.thinking` at runtime before each real phase it launches.

That means these phases can each run with different models:
- `planner`
- `tester`
- `worker`
- `reviewer`
- `chief`

### TDD phased execution

When `tdd: true`, the system runs **four explicit phases** per ticket:

1. **Tester phase** — writes failing tests and produces `tickets/<id>-tester-notes.md`.
2. **Worker phase** — reads tester notes and implements the change.
3. **Reviewer phase** — reviews the result and writes `tickets/<id>-reviewer-notes.md`.
4. **Chief phase** — writes feature memory + worker context and closes the ticket.

This lets you use a lighter, faster model for the tester, a stronger coding model for the worker, and a separate review/finalization model for reviewer/chief:

```json
{
  "tdd": true,
  "agents": {
    "tester": {
      "model": "anthropic/claude-haiku-4",
      "thinking": "off"
    },
    "worker": {
      "model": "anthropic/claude-sonnet-4",
      "thinking": "medium"
    },
    "reviewer": {
      "model": "openai/gpt-4.1"
    }
  }
}
```

When TDD is disabled, the flow starts at **Worker**, then still advances through **Reviewer** and **Chief** as separate runtime phases.

## Master spec model

`01-master-spec.md` is the main planning document. Read it directly — there are no authoring skills to route through.


- Simple feature → lean execution plan, 1–3 thin tickets
- Medium feature → full execution plan with sequenced tickets and validation strategy
- Technically complex feature → execution plan may pause and request additional detail before tickets are generated

### Planning behavior

The planner classifies the spec scope and may emit `BLOCKED` if critical technical detail is genuinely missing. When that happens, add the required context and run `/plan-feature <feature>` again to resume.

## Profiles

Profiles let you choose different agent/model preferences per ticket.

The planner assigns a `- Profile:` to each ticket so mixed frontend/backend features can execute correctly.

Example:
- `frontend` profile for UI and onboarding work
- `backend` profile for API, queue, and DB work
- `direct-mode` profile when you want no subagent delegation

The selected profile for each ticket is read from the ticket file.

## Suggested workflow

1. Run `/init-feature <feature>` to scaffold a stub `01-master-spec.md`.
2. Fill in the spec document.
3. Run `/plan-feature <feature>` to generate the execution plan and tickets.
4. Run `/ticket-validate <feature>` to check for structural issues.
5. Run `/start-feature <feature>` or `/next-ticket <feature>` to begin implementation.
6. If execution stops because a ticket is blocked or needs fix, use `/ticket-blocked`, `/ticket-needs-fix`, `/ticket-done`, `/start-feature`, or `/next-ticket` to steer it.

## Validation rules

| Code | Severity | Description |
|------|----------|-------------|
| `missing-spec-file` | error | A required spec file is missing. |
| `missing-tickets-dir` | error | The tickets directory doesn't exist. |
| `no-tickets` | error | No `.md` ticket files found. |
| `duplicate-ticket-id` | error | Two tickets share the same id (case-insensitive). |
| `missing-dependency` | error | A ticket depends on a non-existent ticket. |
| `dependency-cycle` | error | Circular dependency detected. |
| `invalid-ticket-id` | warning | Ticket id doesn't match recommended pattern (`STK-001` or `T1`). |
| `duplicate-dependency` | warning | The same dependency is listed twice for one ticket. |
| `orphan-ticket` | warning | A ticket has no dependencies and nothing depends on it. |
| `missing-ticket-profile` | error | A ticket is missing a required `- Profile:` line. |
| `ticket-template-mismatch` | error | A ticket does not follow the required ticket markdown template. |
| `execution-plan-template-mismatch` | error | `02-execution-plan.md` does not follow the required execution plan template. |

## Agent coordination

### TDD two-phase execution

When `tdd: true`, each ticket runs through **two separate agent phases**:

```
/next-ticket
  │
  ├─ Phase 1: Tester agent
  │   - Writes failing tests
  │   - Produces tickets/<id>-tester-notes.md
  │   - Outcome: APPROVED | BLOCKED | NEEDS-FIX
  │
  └─ if APPROVED → Phase 2: Worker agent
      - Worker reads tester notes, implements feature
      - Outcome: APPROVED | BLOCKED | NEEDS-FIX
      │
      └─ if APPROVED → Phase 3: Reviewer agent
          - Reviewer validates against ticket criteria
          - Produces tickets/<id>-reviewer-notes.md
          - Outcome: APPROVED | BLOCKED | NEEDS-FIX
          │
          └─ if APPROVED → Phase 4: Chief agent
              - Chief writes worker context + feature memory
              - Outcome: APPROVED | BLOCKED | NEEDS-FIX
```

This lets you assign different models per role: e.g. a fast model for tester, a stronger coding model for worker, a strict reviewer model, and a cheaper summarization/finalization model for chief.

### Checkpoint persistence

Execution state is persisted to `.pending-execution.json` per feature. If pi restarts mid-execution, the extension detects the checkpoint on `session_start` and prompts you to resume.

### Retry context

When a ticket returns to `needs_fix`, the Chief writes `tickets/<id>-worker-context.md` containing:
- Which files were modified (and their status)
- What the reviewer found
- Continuation notes for the next attempt

On retry, the worker reads this file first — it knows exactly what to fix instead of starting from scratch.

### Cost tracking

Every agent phase records token usage and cost. Run `/feature-cost <name>` to see:

| Ticket | Runs | Cost |
|--------|------|------|
| STK-001 | 2 | $0.0050 |
| STK-002 | 1 | $0.0032 |

Costs are stored in `05-cost.json` and track tester / worker / reviewer / chief phases separately.
